# FERPA Compliance Checklist for edu.cloud Pilot Launch

**Checklist Version:** 1.0  
**Last Updated:** April 7, 2026  
**Compliance Status:** IN PROGRESS (Phase 1 - Pre-Pilot)  
**Target Completion:** 30 days before pilot launch  

---

## Overview

This checklist ensures all FERPA compliance requirements are met before and during pilot launch. It is structured in four phases:
- **Phase 1 (PRE-PILOT):** Critical items required before any institutional data access
- **Phase 2 (PILOT LAUNCH):** Items to complete by pilot start date
- **Phase 3 (PILOT OPERATIONS):** Ongoing compliance during pilot
- **Phase 4 (POST-PILOT):** Extended compliance for production

**Completion is the responsibility of assigned owners; completion evidence required for sign-off.**

---

## PHASE 1: PRE-PILOT (REQUIRED BEFORE DATA ACCESS)

### A. FERPA Regulatory Compliance

- [ ] **A1. FERPA Training**
  - **Description:** All team members with access to education records complete FERPA training
  - **Evidence Required:** Completion certificates from FERPA training program (e.g., ed.gov training, law firm webinar)

- [ ] **A2. FERPA Policy Documentation**
  - **Description:** Document edu.cloud's FERPA compliance policies covering:
    - Role of edu.cloud as data processor (not independent recipient)
    - Data source legitimacy assessment
    - Consent requirements for identifiable education records
    - Data use limitations (education research only)
    - Retention and deletion schedules
  - **Evidence Required:** Written FERPA Policy document (5-10 pages)
  - **Owner:** Privacy Lead + Legal Counsel
  - **Due Date:** 20 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Reference FERPA Mapping Document (Part 5)

- [ ] **A3. FERPA Requirement Assessment**
  - **Description:** Internal assessment that all data sources comply with FERPA requirements:

  - **Due Date:** 20 days before pilot data access

### C. Student Consent & Privacy Notice

- [ ] **C1. Student Consent Form (FERPA-Compliant)**
  - **Description:** Create consent form for students to opt-in to outcome data sharing
  - **Form Must Include:**
    - Clear statement of data being collected (educational outcomes, background)
    - Purpose: "To help [Institution] measure and communicate alumni success"
    - Recipient: "[Institution Name] will share your information with edu.cloud"
    - Data use: "edu.cloud will use your data for impact research and reporting"
    - Student rights: access, correction, withdrawal (with process/timeline)
    - Institution contact for privacy questions (name, email, phone)
    - No coercion language: "Your consent is voluntary and is not a condition of enrollment or services"
    - Signature/date/version control
  - **Evidence Required:** Finalized consent form approved by legal counsel and institution
  - **Owner:** Privacy Lead + Legal Counsel
  - **Due Date:** 25 days before pilot data access
  - **Status:** ⬜ Not Started

- [ ] **C2. Student Consent Collection Method**

    - [ ] Web form on institutional portal (student enters consent)

  - **Evidence Required:** Functional consent collection system; test with sample students
  - **Owner:** Product Lead
  - **Due Date:** 20 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** System must store consent with student ID, date, form version, signature

- [ ] **C3. Student Privacy Notice**

### D. Data Classification & Security

- [ ] **D1. Data Classification Schema**
  - **Description:** Define classification for FERPA vs. non-FERPA data
  - **Classifications:**
    - **FERPA: Educational Records** — Data directly related to a student maintained by institution (names linked to outcomes, grades, etc.)
    - **FERPA: Identifiable by Linking** — Public data that can be matched back to identifiable student
    - **Non-FERPA: Public Data** — Already public; no link to individual student
    - **Non-FERPA: Aggregated/De-identified** — No way to identify individual student
    - **Internal: User/System Data** — Not FERPA regulated (employee accounts, system logs)
  - **Evidence Required:** Written schema document; classification examples for each type
  - **Owner:** Privacy Lead + Engineering Lead
  - **Due Date:** 20 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Each database table should have classification tag

- [ ] **D2. Database Tagging for FERPA Data**
  - **Description:** Tag all database tables/fields containing FERPA-regulated data
  - **Implementation:**
    - [ ] Add `ferpa_regulated: true/false` flag to all tables
    - [ ] Document FERPA classification in data dictionary
    - [ ] Automated scan to identify tables missing classification
    - [ ] Engineering team training on classification meanings
  - **Evidence Required:** Database schema documentation with FERPA tags; scan results
  - **Owner:** Engineering Lead
  - **Due Date:** 20 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Must be complete before access controls implemented

- [ ] **D3. Database Encryption at Rest**
  - **Description:** Enable encryption for databases containing FERPA data
  - **Requirements:**
    - [ ] AWS RDS encryption enabled (AES-256)
    - [ ] Encryption keys stored separately from database credentials
    - [ ] Key rotation procedure documented
    - [ ] Test: decrypt encrypted data to verify encryption works
  - **Evidence Required:** AWS console screenshot showing encryption enabled; key management documentation
  - **Owner:** Infrastructure Lead
  - **Due Date:** 15 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Can be enabled after database creation; requires brief restart

- [ ] **D4. Encryption in Transit (TLS)**
  - **Description:** Ensure all data transmission is encrypted
  - **Requirements:**
    - [ ] All database connections via TLS 1.2+
    - [ ] API endpoints require HTTPS
    - [ ] Web forms use HTTPS
    - [ ] SSL/TLS certificate from trusted CA
    - [ ] Certificate renewal automated
  - **Evidence Required:** SSL test results (e.g., SSLLabs report showing A+ rating)
  - **Owner:** Infrastructure Lead
  - **Due Date:** 15 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Standard for production systems; should already be in place

### E. Access Control & Authorization

- [ ] **E1. Role-Based Access Control (RBAC) Design**
  - **Description:** Define roles and permissions for FERPA data access
  - **Roles to Define:**
    - **Engineer (non-data):** Logs, system monitoring; no PII access
    - **Analyst (research):** De-identified aggregated data only
    - **Operations:** User account/system data; no educational records
    - **Research (limited):** Specific educational record access with approval
    - **Admin (database):** Full database access; requires MFA + approval
  - **Evidence Required:** Written RBAC document with roles, permissions, use cases
  - **Owner:** Security Lead + Engineering Lead
  - **Due Date:** 20 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Principle of least privilege; minimize data access needs

- [ ] **E2. Multi-Factor Authentication (MFA)**
  - **Description:** Require MFA for all FERPA data access
  - **Implementation:**
    - [ ] AWS console requires MFA
    - [ ] Database access requires MFA
    - [ ] SSH/bastion host requires MFA
    - [ ] Web application admin panel requires MFA
  - **Evidence Required:** MFA configuration documentation; list of users with MFA enabled
  - **Owner:** Infrastructure Lead
  - **Due Date:** 15 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Acceptable factors: hardware token, authenticator app, SMS (less preferred)

- [ ] **E3. RBAC Implementation**
  - **Description:** Implement RBAC in all systems accessing FERPA data
  - **Systems to Update:**
    - [ ] AWS IAM (role-based permissions for services)
    - [ ] Database access control (role-based queries)
    - [ ] Web application (admin panel access)
    - [ ] Infrastructure (SSH access to servers)
  - **Evidence Required:** Code changes; documentation of permission checks; test results showing unauthorized access blocked
  - **Owner:** Engineering Lead
  - **Due Date:** 15 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Test by attempting unauthorized access; should be blocked

- [ ] **E4. Access Review Procedure**
  - **Description:** Document procedure for quarterly access reviews
  - **Procedure Must Include:**
    - Timeline: review all access quarterly
    - Process: generate list of current access; compare to approved access matrix
    - Action: remove access no longer needed
    - Approval: manager approval for each change
    - Documentation: record of access reviews retained
  - **Evidence Required:** Written procedure document; schedule for first review
  - **Owner:** Security Lead + Privacy Lead
  - **Due Date:** 20 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** First review scheduled: 30 days after pilot launch

### F. Audit Logging & Monitoring

- [ ] **F1. Database Audit Logging Configuration**
  - **Description:** Enable audit logging for all access to FERPA data
  - **Logging Must Capture:**
    - [ ] User/role executing query
    - [ ] Timestamp of access
    - [ ] Query type (SELECT, UPDATE, DELETE, etc.)
    - [ ] Tables/fields accessed
    - [ ] Number of rows accessed
    - [ ] Query duration (for performance monitoring)
  - **Implementation:**
    - [ ] AWS RDS activity stream enabled
    - [ ] AWS CloudTrail enabled for API calls
    - [ ] Database native audit logging enabled
  - **Evidence Required:** Screenshots of logging configuration; sample log entries
  - **Owner:** Infrastructure Lead
  - **Due Date:** 15 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Logs should be immutable (append-only) and retention ≥3 years for FERPA data

- [ ] **F2. Audit Log Retention & Protection**
  - **Description:** Ensure audit logs are protected from modification/deletion
  - **Requirements:**
    - [ ] Logs stored in separate, immutable storage (append-only)
    - [ ] Logs encrypted at rest
    - [ ] Minimum 3-year retention for FERPA-related logs
    - [ ] Access to logs restricted (only security/compliance staff)
    - [ ] Log deletion/archival procedure documented
  - **Evidence Required:** Log storage configuration; retention policy document
  - **Owner:** Infrastructure Lead
  - **Due Date:** 15 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** CloudTrail S3 bucket with MFA delete recommended

- [ ] **F3. Suspicious Activity Alerting**
  - **Description:** Configure alerts for suspicious data access patterns
  - **Alerts to Set Up:**
    - [ ] Bulk data export (>1000 rows in single query)
    - [ ] After-hours data access (outside business hours)
    - [ ] Unusual database activity (failed login attempts, etc.)
    - [ ] Access from unauthorized IP ranges
    - [ ] Concurrent access by same user from different locations
  - **Evidence Required:** Alert configuration in monitoring system (e.g., CloudWatch); sample alert
  - **Owner:** Security Lead
  - **Due Date:** 15 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Alert recipients: Security Lead + Privacy Lead

### G. Retention & Deletion

- [ ] **G1. Data Retention Schedule**
  - **Description:** Define and document retention policy for each data type
  - **Schedule to Define:**
    - **Educational record data (identifiable):** 3 years post-pilot conclusion
    - **Student consent records:** Lifetime of data retention
    - **Aggregated/de-identified data:** Indefinite
    - **Operational logs:** 90 days
    - **Audit/security logs:** 3 years (for FERPA data access audit trail)
    - **Institutional partnership data:** Per institution's requirement
  - **Evidence Required:** Written retention policy document; approval from institutional partners
  - **Owner:** Privacy Lead
  - **Due Date:** 20 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Institutions must approve retention schedule; document in DUA

- [ ] **G2. Data Deletion Procedure**
  - **Description:** Document procedure for secure data deletion
  - **Procedure Must Include:**
    - Automated identification of expired records
    - Secure deletion (not just "soft delete")
    - Deletion logging for audit trail
    - Verification that deletion completed
    - Regular testing of deletion procedure
  - **Implementation:**
    - [ ] Database job that runs periodically (weekly) to identify expired records
    - [ ] Secure deletion that overwrites data before removing (NIST SP 800-88 guidelines)
    - [ ] Deletion logged with timestamp, user, record count
    - [ ] Deletion verification query (confirm records deleted)
  - **Evidence Required:** Code/SQL for deletion job; deletion log sample; test results
  - **Owner:** Engineering Lead + Privacy Lead
  - **Due Date:** 15 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Do NOT rely on cloud provider's automatic deletion; implement custom deletion logic

- [ ] **G3. Backup & Recovery Data Retention**
  - **Description:** Define retention for backup data containing FERPA information
  - **Requirements:**
    - [ ] Backups encrypted
    - [ ] Backup retention aligned with FERPA data retention (3 years)
    - [ ] Backup deletion procedure documented
    - [ ] Test restore from backup to verify integrity
  - **Evidence Required:** AWS backup policy; backup deletion procedure; restore test results
  - **Owner:** Infrastructure Lead
  - **Due Date:** 15 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Backup retention should match primary data retention; can extend for legal hold

### H. Vendor & Third-Party Management

- [ ] **H1. Vendor Data Processing Agreements (DPAs)**
  - **Description:** Execute Data Processing Agreements with all vendors accessing education records
  - **Vendors Requiring DPA:**
    - [ ] AWS (Data Processing Addendum)
    - [ ] Google Cloud (if used)
    - [ ] Analytics platform (Mixpanel) — if receiving any PII
    - [ ] Email provider (if handling education records)
    - [ ] Other: ____________
  - **DPA Must Include:**
    - Processor obligations (protect data, implement safeguards)
    - Scope of processing (what data, for what purpose)
    - Subprocessor rights (vendor cannot subcontract without approval)
    - Security requirements (encryption, access control)
    - Data location (where data is stored/processed)
    - Breach notification (30-day notification requirement)
    - Audit rights (we can audit vendor's security practices)
    - Data deletion (upon contract termination)
    - Liability/indemnification
  - **Evidence Required:** Signed DPA from each vendor
  - **Owner:** Privacy Lead + Procurement
  - **Due Date:** 20 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** AWS DPA is standard; Google Cloud available via console; others require negotiation

- [ ] **H2. Vendor Security Assessment**
  - **Description:** Assess security of vendors accessing FERPA data
  - **Assessment Should Cover:**
    - SOC 2 Type II certification (for cloud vendors)
    - Encryption capabilities
    - Audit logging/monitoring
    - Incident response procedures
    - Security certifications (ISO 27001, FedRAMP, etc.)
    - Subprocessor security practices
  - **Evidence Required:** Vendor security questionnaire responses; copies of security certifications
  - **Owner:** Security Lead
  - **Due Date:** 20 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Use industry-standard security questionnaire (CAIQ or similar)

- [ ] **H3. Vendor Data Minimization**
  - **Description:** Limit data shared with vendors to minimum necessary
  - **Guidance:**
    - [ ] Analytics platforms (Mixpanel): do NOT send education records or student names
    - [ ] Email service (SendGrid): do NOT send education records
    - [ ] Infrastructure (AWS): may receive education records (encrypted, need DPA)
    - [ ] Document vendors and data types they receive
  - **Evidence Required:** Vendor data matrix (vendors vs. data types) approved by Privacy Lead
  - **Owner:** Product Lead + Privacy Lead
  - **Due Date:** 15 days before pilot data access
  - **Status:** ⬜ Not Started
  - **Notes:** Quarterly audit of vendor data access to ensure data minimization maintained

## PHASE 4: POST-PILOT (PRODUCTION & LONG-TERM)

### A. Extended Compliance

- [ ] **P4-A1. Third-Party Security Audit**
  - **Description:** Commission independent security audit of FERPA compliance
  - **Audit Should Cover:**
    - Technical controls (encryption, access control, logging)
    - Governance (DUAs, consent, retention policies)
    - Vendor management (DPAs, security assessments)
    - Incident response procedures
  - **Evidence Required:** Audit report from independent firm; remediation plan for findings
  - **Owner:** CEO + Security Lead
  - **Due Date:** 6 months post-pilot launch
  - **Status:** ⬜ Not Started
  - **Notes:** Budget $15-25K for independent audit; choose firm with higher ed experience

- [ ] **P4-A2. Annual Compliance Attestation**
